# 风铃渠道设计

风铃是自有移动端主客户端，承担产品最完整的语音、多模态、附件和状态体验。

## 风铃需要承载的体验

- 高频语音和文字混合输入。
- 明确区分文本回复与附件产物。
- 对长任务提供即时反馈、进度感知和异步回报入口。
- 让用户在移动端场景下低摩擦发起、查看和继续任务。

## 风铃与 runtime 的关系

- 风铃负责输入输出、展示和交互状态。
- runtime 负责理解目标、推进任务和调度能力。
- 风铃不拥有独立的任务模型或记忆体系。

## 关键交互对象

| 对象 | 风铃中的表现 |
|------|--------------|
| 对话 | 适合快速阅读和追问的文本流 |
| 附件 | 代码块、命令、文档、链接、截图等独立承载 |
| 任务 | 可被发起、查看状态、继续推进 |
| 确认 | 在高风险动作前明确展示范围与后果 |
| 回报 | 已完成、等待中、卡住、需要确认 |

## 设计要求

1. 风铃支持流式反馈。
2. 附件不得被当作纯朗读文本处理。
3. 语音输入和文字输入可以共享同一任务对象。
4. 风铃发起的任务必须能按需投递到 Telegram 等外部集成，但不要求外部渠道承担与风铃对等的完整体验。
