# 测试分型

AI 原生开发不应该把所有验证都塞进 `implementation/tests/`。不同类型的验证有不同职责。

## 1. Deterministic Tests

适合验证确定性、可精确断言的行为：

- 纯函数
- 状态机流转
- 事件分发
- schema 合法性
- 委托边界硬约束
- 路径与协议处理

这些场景不应交给 LLM judge。

## 2. Contract Tests

适合验证接口契约稳定性：

- 任务对象
- 事件对象
- 工具协议
- 渠道负载格式
- evidence trace 结构

## 3. Scenario Evals

适合验证面向用户任务的闭环场景：

- 任务续作
- 多渠道连续性
- 风铃的文本/附件分区
- 提醒与异步回报

这类验证通常从 `requirements/acceptance/` 中读取场景定义。

## 4. LLM-Judge Evals

只用于“语义是否正确但难以硬编码”的部分，例如：

- 澄清问题是否抓住缺失信息
- 最终解释是否覆盖关键点
- 回复是否符合指定交互原则
- 续作说明是否真正利用了已有证据

不应用 LLM judge 来代替硬约束检查。

## 5. Regression Cases

来自真实失败样本，例如：

- 误执行
- 该确认却未确认
- 记忆写错
- 产物丢失
- 渠道回报错误

回归样本优先级高于想象中的边角 case。

## 组织原则

1. 需求层先定义场景和通过标准。
2. 设计层定义如何评和看什么证据。
3. 实现层只负责运行、收集证据和生成报告。
4. 先 deterministic，再 LLM judge。
5. 同一个高价值 case 应支持重复运行和长期回归。
