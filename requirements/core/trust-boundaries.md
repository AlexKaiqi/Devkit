# 功能规格：信任模型与委托边界

## 背景

个人 AI 管家如果要从"回答问题"走向"替用户做事"，就必须回答一个更核心的问题：**什么事情可以自动做，什么事情必须先问，什么事情默认不能做。**

没有信任模型，系统要么过于保守，什么都不敢执行；要么过于激进，造成误执行、越权操作和用户不信任。信任模型的目标，是在"高自主"和"可控"之间建立清晰边界，并把这些边界落实到产品交互、Agent prompt、工具调用和审计日志中。

这里需要区分三个层次：

- **信任模型**：产品层原则，定义什么是高风险、什么必须确认
- **委托边界**：用户、任务、目录、外部系统上的具体授权范围
- **执行前检查**：真正调用工具前，由 runtime 做的最后一道强制检查

## 需求

### 风险分级

所有动作至少分为四级风险：

| 风险级别 | 含义 | 典型动作 | 默认策略 |
|----------|------|----------|----------|
| `L0` 只读低风险 | 读取、总结、查询，不产生副作用 | 读文件、搜索、总结邮件、查行情 | 自动执行 |
| `L1` 本地可逆修改 | 仅修改本地状态，且可回滚 | 写草稿、生成文件、改本地代码未提交 | 自动执行，但需可审计 |
| `L2` 外部或不可逆副作用 | 会影响外部世界，或难以回滚 | 发消息、发邮件、提交代码、远程写入 | 必须确认 |
| `L3` 高敏感操作 | 涉及账户、金钱、隐私、系统安全 | 支付、改密钥、删除大量数据、授权外部系统 | 必须逐次确认，默认保守拒绝 |

### 动作分类

为了让确认策略稳定，需要按动作类型建模，而不是只靠模型自由发挥。

最低应覆盖：

- 本地读取
- 本地写入
- 代码变更
- 版本控制
- 外发通信
- 外部系统写入
- 凭据或权限操作
- 金钱与账户相关操作

### 委托边界

委托边界不是一句抽象的"请谨慎操作"，而是一组会在执行前被检查的具体规则。

它至少应覆盖：

- 哪些目录 / 仓库可读、可写、可提交、可推送
- 哪些外部系统只允许查询，不允许改写
- 哪些联系人 / 渠道允许起草，不允许代发
- 哪些高风险动作永远不能默认执行
- 某次任务中是否拿到了额外的临时授权

委托边界既包含：

- **静态边界**：长期稳定规则
- **动态边界**：本任务、本会话、本次确认新增的允许范围

### 执行前强制检查

委托边界不能只作为 prompt 前提存在，否则很容易被长上下文或多步推理忽略。

因此至少要有两道检查：

1. **规划阶段检查**：让模型在规划时尽量避开明显越界动作
2. **执行前检查**：在工具真正执行前，由 runtime 强制判断是否允许

第二道检查更关键，因为模型只能"提出要做什么"，不能拥有最终执行权。

如果执行前检查不通过，系统应：

- 拒绝执行
- 或转为 `needs_confirmation`
- 或要求用户缩小范围 / 明确授权

### 默认策略

#### 可自动执行

满足以下条件的动作可自动执行：

- 用户明确提出请求
- 风险不高于 `L1`
- 范围清晰、目标明确
- 失败后可解释，可重试，可回滚

示例：

- "看一下这个目录结构"
- "帮我总结这份文档"
- "把这个草稿整理成要点"
- "在本地新建一个临时文件保存结果"

#### 必须确认

满足以下任一条件的动作，必须先确认：

- 风险等级为 `L2` 或 `L3`
- 作用于外部系统
- 会覆盖、删除、推送、发送、发布
- 需要消耗账号额度、访问敏感信息或影响真实世界状态

确认信息至少应说明：

1. 要做什么
2. 影响范围
3. 为什么需要这样做
4. 用户回复什么才视为同意

#### 永不默认执行

以下行为不能仅凭模型自主判断直接执行：

- 未经用户授权主动联系外部人员
- 未限定范围的高危 shell 操作
- 批量删除或覆盖未知范围文件
- 处理金钱或账户安全的最终决策
- 读取或暴露与当前任务无关的敏感内容

### 确认粒度

确认不应一刀切。产品需要支持以下粒度：

- **单次确认**：只允许当前动作执行一次
- **任务级确认**：允许当前任务中的同类动作继续执行
- **会话级确认**：仅适用于低到中风险的重复动作，不适用于 `L3`

V1 默认至少支持**单次确认**，其他粒度可后续演进。

### 最后一步确认

对于 `L2` / `L3` 动作，确认不只是"会话一开始读过规则"即可，而应是执行前的最后一步显式放行。

这意味着：

- 模型已经规划好动作也不代表可以直接执行
- 即使同一轮对话前文已经提过限制，执行前仍要核对当前委托边界
- 确认结果必须能和具体动作、具体范围绑定，而不是一个模糊的"都可以"

### 模糊场景处理

当系统无法确定动作边界时，默认策略是：

1. 不执行
2. 先澄清
3. 把自己理解到的动作范围说出来

例如：

- "帮我清理一下项目" → 先问清理哪些文件
- "把这个发出去" → 先问发给谁、通过什么渠道
- "帮我处理一下 Git" → 先问是提交、推送、回滚还是建分支

### 审计要求

所有高风险动作都必须留下审计痕迹，至少包括：

- 谁发起的
- 系统如何判断风险等级
- 是否经过确认
- 实际执行了什么
- 成功或失败结果

### 用户体验要求

确认不能只是一句"要不要继续"。它应当帮助用户快速判断，而不是增加认知负担。

推荐格式：

- 动作：将要执行的具体操作
- 范围：涉及哪些文件、服务、对象
- 风险：为什么需要确认
- 选项：用户如何批准、修改范围、或取消

## 验收标准

- [ ] 所有工具调用都能映射到明确的风险等级
- [ ] `L0` / `L1` 动作无需额外确认即可执行
- [ ] `L2` / `L3` 动作在执行前必须产生确认节点
- [ ] 未经确认不能自动发送消息、提交代码、远程写入或破坏性删除
- [ ] 所有高风险动作都记录确认与执行审计
- [ ] 当动作范围不清时，系统默认澄清而不是猜测执行
- [ ] 高风险工具调用在真正执行前会再次经过委托边界检查，而不只是依赖 prompt 前提

## 技术约束

- 风险判断不能完全依赖模型自由发挥，应由工具层、委托边界和 runtime 共同约束
- 单一工具内可能同时包含不同风险动作，必要时需在工具参数级别进一步细分
- 信任模型要与任务状态机联动：需要确认的任务必须进入 `needs_confirmation`
- 审计日志至少要能重建"确认前"和"确认后"两个关键节点
- 委托边界不能只放在系统 prompt 里，必须有执行前可检查的表示形式
