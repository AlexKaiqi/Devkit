# 功能规格：记忆层与知识层

## 背景

对个人 AI 管家 / 数字分身来说，最大的难点往往不是"能不能调用工具"，而是**能不能把长期有价值的信息持续积累下来，并在未来正确取用。**

如果没有稳定的记忆与知识构建机制，系统会出现几个典型问题：

- 每次对话都像第一次见面，长期偏好无法延续
- 做过的任务、踩过的坑、形成的结论无法沉淀
- 文档、附件、决策、日志只是堆积，不能形成可复用知识
- 记忆写得很多，但真假混杂、没有来源、无法维护

因此，记忆与知识不是附属功能，而是个人分身成立的核心条件。

## 需求

### 记忆层与知识层的边界

逻辑上，系统应把**记忆层**和**知识层**区分开来；物理上，它们可以共用一个后端系统，也可以分别实现。

- **记忆层**：低成本沉淀、文档友好、适合追加和人工维护
- **知识层**：结构化、可关联、可查询、可维护，适合实体关系、图库、图谱和长期知识资产

当前项目优先定义这两层的**能力边界与交互契约**，不把知识层实现方式绑定为文件。

系统需要明确区分以下几类信息：

| 类型 | 作用 | 典型内容 | 存放位置建议 |
|------|------|----------|--------------|
| 对话上下文 | 支撑当前 turn 或短期连续对话 | 当前问题、最近几轮回复 | 会话历史 / 运行时内存 |
| 任务状态 | 让任务跨等待、跨渠道继续推进 | `task_id`、状态、下一步动作 | 任务状态日志 / 任务注册表 |
| 长期记忆 | 与用户本人和长期偏好相关 | 偏好、习惯、约束、重要背景 | `implementation/assets/persona/MEMORY.md` |
| 工作日志 | 记录最近发生的事实与操作 | 今日完成项、进行中事项、失败记录 | `implementation/assets/persona/memory/YYYY-MM-DD.md` |
| 知识层资产 | 可复用、可查询、可维护的知识资产 | 联系人、资产、项目知识、领域知识、图库、图谱 | 外部知识系统 / 独立知识服务 |
| 决策与经验 | 稳定结论与教训 | 技术决策、踩坑、原则 | `design/decisions/README.md` |

核心原则：**不是所有信息都写进长期记忆。**

### 记忆层写入策略

系统需要定义什么信息值得沉淀，什么不该写入。

#### 应写入记忆层的内容

- 用户长期稳定的偏好与约束
- 未来还会重复使用的重要背景
- 跨任务可复用的事实、经验、决策
- 长周期待跟进的信息
- 对某个专业领域已经验证过的结论

#### 不应直接写入记忆层的内容

- 单次对话的琐碎细节
- 未验证的猜测
- 情绪化、一次性的表达
- 与未来复用无关的中间噪音
- 没有来源的总结性判断

### 知识层预期能力

知识层不要求在本项目内完成实现，但应具备明确的预期能力：

- 支持结构化对象、实体关系和跨对象关联
- 支持来源追踪、更新时间、验证状态
- 支持从记忆层和任务产物中提炼高价值知识
- 支持去重、合并、失效标记和版本演化
- 支持按任务场景召回相关知识，而不是只做全文堆积
- 支持非文本资产，如图片、附件、图库元数据

### 来源与可追溯性

任何长期记忆或知识条目，理想上都应回答三个问题：

1. 这条信息从哪里来
2. 它有多可信
3. 未来何时该复查或更新

V1 不要求所有知识都严格结构化，但至少应具备：

- 来源指向：用户明确告诉、任务执行结果、文档提炼、工具查询结果
- 最近更新时间
- 是否已验证

### 记忆层到知识层的晋升

记忆层和知识层应通过一个明确的"提炼 / 晋升"流程连接：

1. 先把原始、有潜在价值的信息写入记忆层或工作日志
2. 再从中筛选出稳定、可复用、值得结构化维护的内容
3. 将这些内容晋升到知识层，而不是长期停留在原始文档中

这意味着：

- 不是所有记忆都会变成知识
- 不是所有知识都需要直接从对话中生成
- 记忆层更偏原始沉淀，知识层更偏长期资产

### 记忆构建流程

记忆与知识的最小闭环应是：

1. **捕获**：从对话、任务结果、工具输出中识别潜在有价值信息
2. **筛选**：判断是否值得长期沉淀
3. **结构化**：放入正确的层级和载体，而不是全部塞进一个地方
4. **关联**：与用户、项目、任务、领域知识建立关联
5. **复用**：在后续任务中被正确召回
6. **维护**：定期合并、修正、去重、淘汰过期内容

### 知识维护要求

知识层不是 append-only 垃圾堆，需要维护机制。

最低要求：

- 去重：相同事实不应在多个位置不断重复堆积
- 合并：零散日志要能沉淀为稳定知识
- 标记失效：过时知识不能长期伪装成事实
- 分层：偏好、事实、经验、结构化知识要分开维护
- 可编辑：用户和 Agent 都能安全修改

### 召回策略

知识价值不在于"写下来"，而在于"用得上"。

系统应优先支持：

- 会话开始时读取长期记忆
- 执行特定任务时按领域读取相关知识
- 遇到相似问题时优先检索已有结论
- 在回答中显式利用历史偏好与上下文

### 产品体验要求

从用户视角，记忆与知识能力至少应表现为：

- 不需要反复解释自己是谁、偏好是什么
- 以前做过的事情能被延续，而不是重新摸索
- 重要事实会被记住，错误事实能被纠正
- 随着使用时间增长，系统越来越像"熟悉我的分身"

## 验收标准

- [ ] 系统明确区分对话上下文、任务状态、记忆层、知识层和决策经验
- [ ] 记忆层写入有筛选规则，而不是把所有聊天内容直接追加
- [ ] 长期记忆条目可追溯来源，至少区分"用户提供"、"任务产出"、"工具查询"
- [ ] 日志、记忆层、知识层三层信息可以相互转化，而不是彼此割裂
- [ ] 记忆错误能被修正，过时知识能被标记或淘汰
- [ ] 后续任务能实际复用先前积累的记忆或知识

## 技术约束

- V1 可以继续用文件承载记忆层，但不应把知识层实现永久绑定为文件系统
- 记忆系统不能完全依赖模型"自己判断"，需要文档规则和结构约定共同约束
- 知识层的实现可由外部 agent 管理的独立系统承担；本项目当前只定义能力契约
- 结构化知识与自由文本记忆应并存，不能只押注单一格式
- 维护流程应尽量低成本，否则知识库会迅速失控
