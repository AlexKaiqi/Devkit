# 功能规格：任务生命周期与续作

## 背景

如果所有请求都只停留在聊天消息里，AI 管家很快会退化成普通聊天助手。对个人分身产品来说，真正重要的是：用户交代一件事后，系统能否把它视为一个持续推进的任务，并在执行、等待、完成、失败这些阶段里给出稳定反馈。

当前产品已经具备对话、工具调用、事件驱动定时器和多渠道回报能力，但"任务"还没有被显式建模为一等对象。需要一份统一的任务生命周期规范，作为后续 UI、审计、Agent prompt 和异步编排的共同基础。

## 需求

### 任务是一等对象

任何"去做一件事"的请求，都应被系统抽象成任务，而不是只留下自然语言回复。

最小任务字段建议包括：

- `task_id`：全局唯一标识
- `session_key`：关联会话
- `source_channel`：风铃 / Telegram / 其他
- `title`：用户可读的一句话标题
- `intent`：任务意图摘要
- `risk_level`：风险等级
- `state`：当前状态
- `created_at` / `updated_at`
- `artifacts`：代码、文档、截图、链接等结果附件
- `next_action`：下一步动作或等待条件
- `continuation_of`：如果这是续作任务，指向前一个任务

### 状态模型

任务状态统一定义如下：

| 状态 | 含义 | 用户是否可见 | 典型触发 |
|------|------|--------------|----------|
| `needs_clarification` | 目标不清，缺少关键信息 | 是 | 用户需求过于模糊、缺少范围 |
| `needs_confirmation` | 已理解任务，但涉及高风险动作，需先确认 | 是 | 删除、提交、外发、远程写入 |
| `queued` | 已接受任务，等待执行资源或前置依赖 | 是 | 子任务排队、等待上游结果 |
| `running` | 正在执行 | 是 | 已开始工具调用、分析、编排 |
| `waiting_external` | 等待定时器、网络回调、外部服务或异步结果 | 是 | 定时提醒、长任务后台处理 |
| `waiting_user` | 需要用户补充信息后才能继续 | 是 | 缺少账号、路径、确认内容 |
| `completed` | 已完成，且结果已可回报 | 是 | 任务闭环结束 |
| `failed` | 执行失败，且需要用户知晓原因或处理 | 是 | 工具失败、依赖不可用、权限不足 |
| `cancelled` | 任务被用户或系统取消 | 是 | 用户撤销、上游任务失效 |

### 状态流转规则

任务的核心流转应满足：

1. 用户发起后，任务必须进入上述状态之一，不能停留在"无状态完成"
2. 高风险任务必须先进入 `needs_confirmation`，确认后才能进入 `queued` 或 `running`
3. 需要更多上下文的任务进入 `needs_clarification` 或 `waiting_user`
4. 进入异步世界的任务必须进入 `waiting_external`
5. 所有终态只允许是 `completed`、`failed`、`cancelled`

推荐流转示例：

```text
用户请求
  ├─ 信息不足 → needs_clarification
  ├─ 风险较高 → needs_confirmation
  ├─ 可直接执行 → running
  └─ 需要排队/依赖 → queued

running
  ├─ 等待外部事件 → waiting_external
  ├─ 等待用户输入 → waiting_user
  ├─ 成功结束 → completed
  ├─ 失败结束 → failed
  └─ 用户撤销 → cancelled
```

### 回报规则

任务不是只在结束时才需要出现。系统至少应在以下时点向用户反馈：

- 任务已被接受
- 任务需要澄清或确认
- 任务进入长时间等待
- 任务完成
- 任务失败

对用户的回报格式应尽量稳定：

1. 当前状态
2. 已做了什么
3. 结果或卡点
4. 是否需要用户继续操作

### 附件与结果分离

任务完成或阶段性回报时，正文与附件应分离：

- 正文：结论、摘要、下一步建议
- 附件：代码 diff、命令、日志片段、文档、图片、链接

### 跨渠道连续性

任务属于用户，不属于单一聊天窗口。

因此需要满足：

- 风铃发起的任务，可按配置投递到 Telegram 等外部渠道
- 外部渠道若承接回报，应保留 `task_id` 并能回指风铃中的任务状态
- 同一 `session_key` 下，任务状态能被持续恢复

### 与事件系统的关系

所有异步任务都应通过事件系统恢复，而不是靠阻塞等待。

典型模式：

- 定时提醒：`running` → `waiting_external` → `completed`
- 后台长任务：`running` → `waiting_external` → `running` → `completed`
- 依赖外部授权：`running` → `waiting_user` → `running`

详见 [event-system.md](event-system.md)。

### 任务续作与恢复

这里的"恢复"不应理解为把上一段完整对话上下文原样装回模型，而更像是：

**基于一个做了一半的旧任务，生成一个可继续推进的新任务。**

因此续作任务至少应利用：

- 旧任务目标
- 当前状态与卡点
- 已产生的附件和中间结果
- 审计日志中的关键执行记录
- 和记忆层 / 知识层相关的上下文

续作的重点不是"复原整段历史"，而是为下一步动作组装一个足够好的继续执行输入包。

典型场景：

- 长任务中断后继续推进
- 用户隔了一段时间回来问"上次做到哪了，继续"
- 外部依赖恢复可用后，从失败任务继续
- 需要把同一目标拆成多个连续任务时

## 验收标准

- [ ] 任意长任务或异步任务都会生成 `task_id`
- [ ] 每个任务始终处于定义好的状态之一
- [ ] 所有状态迁移都带时间戳，并可按 `session_key` 追溯
- [ ] `needs_confirmation` 状态的任务在确认前不能执行高风险动作
- [ ] `waiting_external` 状态的任务可被事件恢复并继续推进
- [ ] `completed` / `failed` 任务都能给出稳定的用户回报格式
- [ ] 任务结果支持正文与附件分离展示
- [ ] 半途任务可以基于状态、审计和产物生成续作任务，而不依赖完整对话历史重放

## 技术约束

- V1 可先用文件或 JSONL 审计补足任务状态流转，不必立即引入数据库
- 状态模型必须与 `session_key` 对齐，否则异步结果无法正确投递
- 任务状态更新不能阻塞主对话响应
- 子任务、并行任务、重试策略可以后续扩展，但不能破坏主状态机
- 任务续作依赖状态、产物和审计，而不是要求完整上下文永久保留
