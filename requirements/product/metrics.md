# 产品衡量

本文定义以风铃为主入口、共享 AI 分身 runtime 为底座的产品体系的核心衡量方式。目标不是统计“聊了多少句”，而是衡量它是否真的替用户完成了事情。

## 衡量原则

- **先看闭环，再看功能数**：能力多不代表产品成立，关键是任务有没有被推进并回报
- **先看体验，再看模型分数**：用户感知的是延迟、可靠性、可控性，不是 benchmark
- **先看信任，再看自治**：越像分身，越要可解释、可确认、可回滚

## 北极星指标

### 每周闭环任务数

定义：用户发起后，系统完成执行或明确给出结果，并成功回报给用户的任务数。

一个任务被视为闭环，至少满足：

- 有清晰的发起事件
- 有执行过程或异步等待
- 有最终结果、失败说明或后续动作建议
- 结果成功送达用户

这是最能体现"分身价值"的指标。

### 有效代理率

定义：所有任务请求中，最终**不需要用户手工补做核心步骤**的比例。

示例：

- "10 分钟后提醒我" → 到时成功提醒，算有效代理
- "帮我查某文件并总结" → 自动完成并给出结果，算有效代理
- "帮我改代码" → 只回答思路，没实际执行，不算有效代理

## 核心体验指标

### 首响应延迟

衡量从用户发起请求到收到首个可见反馈的时间。

建议目标：

- 文字输入：p50 < 2.5s，p95 < 6s
- 语音输入：松手到首个文本反馈 p50 < 3.5s

### 任务启动确认时延

衡量需要较长执行时间的任务，系统多久能先告诉用户"我已开始处理"。

建议目标：

- p50 < 2s
- p95 < 5s

### 异步结果回报时效

衡量定时器、后台任务、等待外部条件的任务，是否能在承诺时点附近回报。

建议目标：

- 定时任务准时送达率 > 99%
- 计划触发时间与实际触达时间偏差的 p95 < 10s

### 多渠道连续性

衡量任务是否能在不同渠道连续完成。

建议目标：

- 桌面端发起、Telegram 收到结果的成功率 > 95%
- Telegram 发起、桌面端可追溯上下文的成功率 > 95%

## 工具与执行指标

### 工具调用成功率

定义：模型已决定调用工具的请求中，首轮工具执行成功并进入下一步的比例。

建议拆分统计：

- `exec`
- `read_file`
- `write_file`
- `search`
- 外部 MCP / CLI

### 任务完成率

定义：已进入执行态的任务中，最终到达"完成"或"明确失败说明"的比例。

目标不是只追求完成，而是减少"悄悄卡住"。

### 平均工具轮数

衡量一个任务平均要经历多少轮 tool-calling 才能得到结果。

这个指标越稳定，说明：

- prompt 设计更清晰
- 工具定义更准确
- 模型对任务编排更成熟

## 信任与可控指标

### 确认覆盖率

定义：所有高风险操作中，先征得用户确认的比例。

要求：

- 破坏性操作确认覆盖率 = 100%
- 对外发送操作确认覆盖率 = 100%

### 误执行率

定义：执行了不该执行的工具调用，或执行范围超出用户意图的比例。

这是比"工具调用成功率"更重要的安全指标。

### 失败可解释率

定义：失败任务中，系统是否明确说明失败原因、卡点位置和下一步建议。

建议目标：

- > 90% 的失败任务能给出可操作解释

### 记忆纠错率

定义：被用户明确纠正的长期记忆条目占比。

该指标偏高时，通常说明：

- 记忆写入策略过于激进
- 事实与偏好没有分层
- 缺少用户确认或记忆来源标注

## 记忆层与知识层指标

### 记忆复用率

定义：任务或对话中，系统实际用到了先前沉淀的长期记忆或知识条目的比例。

这个指标偏低，通常意味着：

- 写下来的内容没有被召回
- 知识组织方式不利于检索
- 记忆和任务场景没有建立关联

### 知识沉淀率

定义：任务执行后，真正被转化为可复用知识资产的结果占比。

例如：

- 一次代码修复沉淀成踩坑记录
- 一次用户澄清沉淀成长期偏好
- 一次研究结果沉淀成结构化知识

如果任务做完后只留下一段聊天文本，而没有进入长期可用层，这个指标会偏低。

### 知识维护负债

定义：知识库中已知存在但未处理的问题数量，例如重复条目、过时条目、无来源条目、互相矛盾条目。

这个指标越高，说明知识库越接近"不可维护状态"。

### 来源覆盖率

定义：长期记忆或知识条目中，带有来源说明或可追溯出处的比例。

建议目标：

- 重要长期记忆来源覆盖率 > 90%
- 高价值知识条目来源覆盖率 > 95%

## 产品阶段目标

### V1：可信可用

重点验证：

- 风铃和 Telegram 双入口可稳定使用
- 文字 / 语音交互自然
- 异步提醒和任务回报可靠
- 本地工具调用可闭环

### V2：持续代理

重点验证：

- 任务状态可见
- 失败能恢复
- 记忆质量提升
- 知识沉淀与复用开始形成正循环
- 能稳定处理多步任务

### V3：高自主但可控

重点验证：

- 主动推进的价值是否大于打扰
- 多领域任务是否可统一编排
- 用户是否愿意把更多真实事务交给系统

## 数据来源建议

当前可直接利用的数据源：

- `implementation/data/voice-audit/`：请求时间、响应时间、渠道、输入类型
- EventBus / Timer API 日志：定时任务创建、触发、投递
- 服务日志：工具执行失败、外部依赖异常、重试情况
- `implementation/assets/persona/MEMORY.md` / `implementation/assets/persona/memory/`：长期记忆与工作日志
- `design/decisions/README.md` / `implementation/data/`：可复用知识资产

后续建议补充：

- 任务 ID 与状态流转日志
- 工具调用明细审计
- 失败原因分类
- 用户确认事件日志
- 记忆写入事件日志
- 知识条目来源与更新时间

## 评审节奏

建议每周至少做一次产品回顾，按下面顺序看：

1. 北极星指标是否上升
2. 延迟和可靠性是否退化
3. 误执行或记忆错误是否增加
4. 本周新增能力有没有真正提升闭环任务数

如果一个新功能不能提升闭环任务数、有效代理率、或关键体验指标，它更像技术展示，而不是产品进展。
